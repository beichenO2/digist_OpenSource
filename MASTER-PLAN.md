# 总纲：多项目协同作战计划

本文档是所有项目助理的唯一参考源。每个项目的助理在启动后必须阅读本文档，理解自己的定位、与其他项目的关系、以及工作规则。

**核心原则：永不停止工作。YOLO 模式。不等用户。不弹 question。一直干到所有 Agent 自然死亡。**

## 一、项目总览

共 5 个项目，由不同的助理各自管理，通过 gsd-2 协调系统实现跨项目协作。

| # | 项目代号 | 一句话描述 | 状态 | 与龙虾的关系 |
|---|---------|-----------|------|------------|
| 1 | **龙虾（Lobster）** | AI 助手主体——融合 OpenClaw+DFlow+CloudCode，具备安全、高效、自动、懂我、主观能动性 | 已有代码基础 | 本体 |
| 2 | **知识杠杆（KnowLever）** | 超大知识库管理——RAG+LLMWiki+Skill蒸馏，处理万字文献/百篇论文/百小时网课 | 从零开始 | 龙虾的知识引擎 |
| 3 | **LLMWiki** | 小专题即时知识整理——可视化静态网站，Mermaid架构图+点击跳转+二级展开 | 已有代码基础 | 龙虾的轻量知识入口 |

| 5 | **自动化办公（AutoOffice）** | PPT/PDF/LaTeX报告生成+文档整理总结+Cursor加速 | 已有代码基础 | 龙虾的输出通道 |

4. **龙虾（Lobster）** — 依赖所有其他项目，先做不依赖外部的部分（安全、长期记忆、柔性规划）

## 三、各项目详细需求

### 3.1 龙虾（Lobster）

**来源**：融合 OpenClaw + DFlow + CloudCode 三个现有项目

**五大核心需求：**

#### A. 安全
1. **龙虾本身不能威胁我**
   - 沙箱隔离：龙虾运行环境与宿主机隔离
   - GitHub 上传安全：开发一个专门的 Skill，每次 git push 前自动审查
     - 检查是否有密钥、token、敏感信息泄露
     - 强制只能上传到用户的 private repo
     - 上传到 public repo 必须手工操作，Agent 没有这个权限
   - 参考用户提供的 Twitter 安全推文（用户会在项目中提供），自主研究安全最佳实践
   - Agent 自己不懂的安全问题要自己去搜索学习，不要问用户
2. **别人不能威胁我的龙虾**
   - 外部接口的访问安全
   - 防止 API 被恶意利用
   - 认证、授权、速率限制

#### B. 高效
1. **工具多、新、有效**
   - Skills 自进化：定时上网搜索最新资讯 → 自动更新整理 → 优化现有 Skills
   - 工具库持续扩充
2. **多 Agent 协作**
   - 使用 gsd-2 系统实现多 Agent 并行工作
   - 龙虾项目会克隆 gsd-2 到自己目录下使用

#### C. 自动
1. **不停**
   - 使用 gsd-2 的 YOLO 模式，永不停止
   - 做完一个任务自动推进下一个
2. **懂我（长期记忆 + 强化学习）**
   - 长期记忆系统：记住用户的偏好、习惯、历史指令
   - 强化学习：根据用户反馈优化输出
   - 目标：用户给更模糊的描述 → 获得更符合偏好的结果
   - 可参考 Lara Croft 主演的 AI 记忆项目（需调研具体项目名）
3. **全流程自动**
   - 开发完自动测试，测试完自动修复，修复完自动验证
   - 不需要用户说"你去做测试"才做测试
   - 完整的 开发→测试→修复→验证→部署 自动流水线

#### D. 主观能动性
1. **主动沟通**
   - 不只是被动回复，要主动发现问题、主动整合关联任务
   - 发现两个历史任务有关联 → 自动整合 → 通知用户
   - 在不停循环中主动寻找可优化的点
2. **柔性规划**
   - 不是刚性的"几点做什么"计划
   - 允许偏差、延误、变更
   - 智能反馈和监督：偏离计划时自动调整而不是失败
   - 根据实际执行情况动态调整后续计划

### 3.2 知识杠杆（KnowLeverage）

**状态**：从零开始

**处理流程：**
```
原始信息（RAW）
  → LLMWiki（中间层，结构化整理）
    → RAG（检索增强生成，管理海量信息）
      → Skill 蒸馏（将知识凝练为可执行的 Skill）
```

**待研究的关键问题：**
1. Skill 蒸馏的语料边界——文本量多大时蒸馏效果开始下降？还是越多越好？
2. Skill 为什么有效——它真的能代表知识背后的思想吗？还是只是模型的幻觉？
3. 如何评估蒸馏后的 Skill 质量？

**应用场景举例：**
- 用户学期货交易网课 → 把老师的教学内容蒸馏成 Skill → 之后用这个 Skill 辅导用户交易
- 用户读几百篇论文 → RAG 管理 → 随时检索 → 核心观点蒸馏成 Skill

**最终形态：** 可视化静态网站（如果可行的话），与 LLMWiki 共用展示层

### 3.3 LLMWiki

**状态**：已有代码基础（有成功的网站经验可复用）

**定位**：即时小专题知识整理，语料量较少但可能逐渐增加

**核心需求：**
1. **可视化静态网站**
   - 复用用户已有的成功网站的风格和经验
   - Mermaid 图展示架构关系
2. **二级展开结构**
   - 每张 Mermaid 图只有两级：主节点 + 叶子节点，没有中间节点
   - 层次复杂时：叶子节点可以点击进入下一层，再展开一张二级图
   - 一层一层往下画，而不是在同一层全部展开（否则字太小、图太大）
3. **术语/变量解释**
   - 专业名词可以点击查看解释
   - 代码变量可以点击查看原代码中的关键部分
   - 不是粘全部代码，是关键片段，帮助理解工作原理
4. **技术栈**
   - Mermaid 图形渲染
   - 静态网站生成
   - 点击跳转/交互

### 3.4 信息获取（Digest）

**状态**：已有代码基础

**三大模块：**

#### A. 工具和渠道
- 更好的爬虫引擎
- OpenCLI 复用浏览器 cookies 获取需要登录才能看到的内容
- 多渠道并行爬取

#### B. 搜索策略（需要深入研究）
- **信息源选择**：怎么选择合适的信息源？什么样的源是高质量的？
- **搜索方式**：关键词搜索 vs 语义搜索 vs 其他方式
- **搜索有效性验证**：
  - 怎么确定关键词能匹配到目标信息？
  - Top 3/Top 10 够了吗？Top 10-20 里会不会有更好的结果被遗漏？
  - 需要建立搜索质量评估机制
- **触发时机**：什么时候发起信息获取？由谁触发？（龙虾触发 / 定时触发 / 事件触发）

#### C. 预处理
- **核心任务**：将各种格式转为 Markdown（大模型只要 Markdown）
- **输入格式**：PDF、音频、图文、网页、视频字幕等
- **图片处理策略（关键决策）**：
  - **不需要图的领域**（金融、电子信息等）：图的信息已在文字中，丢弃图片以降低成本
  - **需要图的领域**（医学等）：保留图片，引入 VLM（视觉语言模型）
  - 问题：PDF/Markdown 整合时图片怎么处理？怎么与文字关联？怎么在 RAG 中检索图片？

### 3.5 自动化办公（AutoOffice）

**状态**：已有代码基础

**三大功能：**

#### A. 报告生成
- PPT 生成（要好看，不能像 ChatGPT 直接生成的那样丑）
- PDF 报告
- Word/Docs 报告
- LaTeX 论文/报告
- **关键要求**：格式正确、排版美观、像人写的、不是 AI 味道

#### B. 整理总结
- 用户通过龙虾发送信息（飞书）→ 龙虾转交自动化办公处理
- 能接收多种输入：
  - 文件（不只是消息文本）
  - 链接（自己去下载）
  - 压缩包（自己解压）
  - 大量语料
- 根据内容量决定处理方式：
  - 内容少 → 交给 LLMWiki 做小专题
  - 内容多 → 交给知识杠杆做大知识库
- 最终输出：Mermaid 架构图 + 可点击展开的可视化页面

#### C. Cursor 加速
- 用 Cursor Agent（gsd-2 方式）加速处理
- 同一对话少开会话多用 tokens → 更便宜
- YOLO 模式全自动处理
- 需要针对文档处理场景特殊化适配 gsd-2 的工作流

## 四、跨项目协同规则

### 4.1 所有项目共用 gsd-2 协调系统

- 协调文件夹：`~/.gsd2/coordination/`
- 每个项目克隆 gsd-2 到自己目录下
- 由最后一个接棒的助理统一创建所有子 Agent + 全局 CLK
- 全局 CLK 每 5 分钟扫描所有项目状态

### 4.2 能力发布与依赖解锁

- 项目完成某个能力（如信息获取完成了爬虫 API）→ 代理验收通过 → 发布到 `capabilities/`
- 等待方的代理在守望循环中自动发现 → 解锁被阻塞的任务
- 全局 CLK 也会扫描并主动通知

### 4.3 项目间代码同步

当一个项目的改动影响到另一个项目时：
1. 改动方的代理在 `messages/` 中发布变更通知
2. 通知中说明：改了什么、影响哪些项目、需要对方做什么适配
3. 受影响方的代理收到通知后，创建适配任务交给主控

### 4.4 工作优先级

每个项目的助理在拆解任务时，遵循以下优先级：
1. **不依赖外部的功能** — 立即开始
2. **被其他项目等待的能力** — 优先完成（全局 CLK 会发提醒）
3. **依赖外部但外部已就绪的功能** — 解锁后立即开始
4. **依赖外部且外部未就绪的功能** — 最后做

## 五、工作模式与规则

### 5.1 YOLO 模式（所有项目统一使用）

- **用户走了，不会回来确认任何事**
- 代理全权负责规划和决策
- 永不弹 question 等用户
- 阶段完成后自己验收、自己推进下一阶段
- 做完所有需求就自动扩展（寻找可以优化、创新的方向）

### 5.2 永不停止

- 这是最重要的原则
- 没有任务 → 找任务（优化、测试、文档、适配其他项目）
- 不确定怎么做 → 自己决定最合理的方案执行，不问用户
- 遇到错误 → 自动重试/修复/绕过
- 做完了 → 继续做更多（优化性能、增加测试、改进文档、适配其他项目、创新功能）

### 5.3 已有代码的处理

大部分项目已经有代码基础。助理启动后：
1. 先扫描项目目录，理解现有代码做了什么
2. 对照本文档的需求清单，标记哪些已实现、哪些未实现
3. 已实现的：
   - 检查质量 → debug/优化/增加测试/verify
   - 做跨项目适配（让其他项目能调用）
   - 寻找进一步创新的空间
4. 未实现的：按本文档要求实现
5. 与本文档需求不一致的：**按本文档的要求来**

### 5.4 全流程自动

每个功能的开发都必须自动走完全流程：
```
设计 → 开发 → 测试 → 修复 → 验证 → 文档 → 适配其他项目
```
不需要用户在中间催促任何一步。

### 5.5 不确定时的决策原则

当遇到不确定的问题时（因为用户不在，没法问）：
1. 先搜索：上网搜索最佳实践
2. 再分析：对比 2-3 种方案的优劣
3. 选最安全/最通用的方案执行
4. 在 `.planning/decisions.md` 中记录决策理由，用户回来后可以审查
5. **绝对不要停下来等用户**

## 七、收尾规则

当所有 Agent 资源耗尽时：
1. **各项目 CLK**：写项目总结报告（各阶段完成状态、Agent 生命周期、健康度评价）
2. **各项目代理**：写工作反思文档（过程问题、改进建议、gsd-2 改进方向）+ HANDOFF.md
3. **全局 CLK**：写跨项目协调总结（哪些依赖解决了、哪些没有、各项目进度对比）

用户醒来后，阅读这些文档即可了解全部情况，再次发送启动 prompt 即可开启新一轮。
