## 概述

项目工程化实践中，参考资料的管理是一个容易被忽视但却非常重要的环节。良好的[[concept-reference-management|参考资料管理]]策略能够帮助团队成员快速定位所需的技术资料，减少重复搜索时间，同时保持项目仓库的整洁和可维护性。

本文档确立的 **reference/** 目录规范是一种简单而有效的[[concept-reference-management|参考资料管理]]方案。该方案的核心思想是将只读性质的参考资料与项目源代码进行物理分离，形成独立的管理区域。这种分离策略具有多重优势：首先，它在视觉上清晰地区分了「我们写的代码」和「别人产出的资料」，避免了文件性质的混淆；其次，它为团队提供了一个集中的资料入口，便于新成员快速了解项目涉及的技术栈和参考来源；最后，它有助于建立项目知识库，形成技术积累。

该规范适用于各种规模的软件开发项目，尤其适合需要频繁查阅技术文档、学术论文、开源项目源码的研发团队。通过遵循这一规范，团队可以更高效地管理和复用技术资料，提升整体研发效率。

## 核心要点

### 目录定位

- **reference/** 是项目中专门用于存放只读参考资料的目录
- 该目录下的所有文件均为参考资料，不包含项目源代码
- 存放的内容属于「外部输入」而非「项目产出」

### 资料类型

根据规范，**reference/** 目录可以包含以下类型的参考资料：

| 资料类型 | 说明 | 示例 |
|---------|------|------|
| 开源项目 | 相关的开源代码仓库或组件 | 第三方库的 GitHub 链接 |
| 学术论文 | 技术相关的论文文献 | 架构设计论文、算法研究 |
| 技术博客 | 高质量的技术文章摘录 | 框架官方文档、性能优化文章 |
| 外部文档 | 第三方工具的使用手册 | API 文档、SDK 说明 |

### 大文件处理策略

对于占用空间较大的参考资料，规范建议采用以下两种处理方式：

**方式一：使用 Git LFS**

Git LFS（Large File Storage）是 Git 的扩展工具，专门用于高效管理仓库中的大文件。通过 LFS 追踪，Git 不会将大文件内容直接存储在仓库中，而是只存储指针引用，实际文件内容存储在 LFS 服务器上。这样可以显著减小仓库体积，同时保持大文件的可访问性。

启用 Git LFS 的基本步骤：

1. 安装 Git LFS：`git lfs install`
2. 追踪大文件类型：`git lfs track "*.zip"`
3. 提交 .gitattributes：`git add .gitattributes && git commit`

**方式二：仅保留链接**

对于确实很大的文件（如视频、完整的 PDF 合集等），推荐的做法是不将文件本身放入仓库，而是仅保留外部链接。这种方式的优势包括：

- 完全避免仓库膨胀
- 用户按需下载
- 链接可指向 CDN 或云存储，确保可用性

### 核心原则

1. **只读性**：存放在 reference/ 中的资料应被视为只读，不应直接在目录内修改
2. **可追溯性**：资料的来源应当清晰可查，便于团队成员溯源
3. **组织性**：建议在目录内部建立分类子目录，如 reference/papers/、reference/blogs/ 等

## 知识结构

### 目的层

**[[concept-reference-management|参考资料管理]]的核心目标是实现技术知识的有效积累和复用**。在软件开发过程中，团队成员经常需要查阅各种外部资料来理解技术方案、解决具体问题。如果没有统一的管理机制，这些资料可能会散落在各个成员的本机或者即时通讯工具中，导致：

- 资料难以共享，新成员无法快速获取相关技术背景
- 相同资料被重复下载，浪费存储空间
- 技术选型决策缺乏可追溯的依据

### 方法层

**建立专门的 reference/ 目录作为参考资料库**，与源代码目录物理分离。这种方法包含以下具体实践：

- 在项目根目录创建 reference/ 子目录
- 按资料类型建立子目录分类（如 papers、blogs、docs、tools）
- 对大文件使用 Git LFS 或外部链接
- 在 README 中说明 reference/ 的用途和维护方式

### 原理层

**物理分离原则是模块化思想在项目文档管理中的应用**。软件工程中的模块化原则不仅适用于代码组织，同样适用于项目文件结构。通过将不同性质的文件（源码、配置、文档、参考资料）分离存放，可以实现：

- **关注点分离**：每类文件各司其职，结构清晰
- **版本控制友好**：不同类型文件的变更频率不同，分离后便于独立管理
- **团队协作效率**：新成员可以通过 reference/ 快速了解项目的技术上下文

## 待探讨

在应用本规范时，以下问题值得团队进一步讨论和明确：

### 目录内部组织

本规范仅定义了 reference/ 目录的存在性，但未指定其内部结构。团队需要根据实际情况决定：

- 是否需要按技术领域分类（frontend/、backend/、devops/）
- 还是按资料类型分类（papers/、blogs/、docs/）
- 是否需要维护一个索引文件说明各资料的用途

### 参考资料的更新机制

随着项目演进，参考资料可能会变得过时。团队应考虑：

- 设立定期审查机制，清理或更新过时资料
- 在何种情况下应移除某份参考资料
- 如何标注资料的时效性和可信度

### 与内部文档的边界

团队需要明确区分存放在 reference/ 的外部资料和项目自身编写的内部文档（通常放在 docs/ 或 wiki/ 中）。建议在项目 README 中明确说明各文档目录的用途和边界。
